**主要问题**
1. 输出合约没有被稳定执行
   skill 要求默认走 `组会分享/<日期>/<论文标题>.md`，且 H1 默认是“组会分享笔记：...”，但当前很多笔记在 `agent skill/<title>/<title>.md` 下，H1 也常常只是论文标题。长标题还被“文件夹名 + 文件名”重复使用，部分路径接近或超过 Windows 传统 260 字符限制，后续批处理、同步、插件扫描都会不稳。

2. 图片资产策略过重
   现在很多笔记末尾都有“MinerU 图片资源完整性索引”，相当于把所有 MinerU 裁剪图都塞进正文。它保证了资产不丢，但让组会笔记变成半个资源清单。比如 [Skill-3D 笔记](<C:/Users/21410/Documents/Obsidian Vault/agent skill/Skill-3D- Evolving Scene-Aware Skills for Agentic 3D Spatial Reasoning/Skill-3D- Evolving Scene-Aware Skills for Agentic 3D Spatial Reasoning.md:195>) 还有 `![[Pasted image ...]]` 这类根目录 Obsidian 粘贴图引用，不够可迁移，也不符合 skill 里相对 `assets/...` 的约定。

3. “按原文顺序”的证据链还不够可验证
   skill 和 `source-order.md` 已经要求建立 Figure/Table/Equation 统一顺序，但当前更像是模型遵循了要求，而不是有强校验。`extract_source_order.py` 是正则扫描，面对 MinerU 的 hash 图片、无 caption 裁剪图、表格截图、附录编号时会比较脆。

4. Review pass 还是软约束
   [SKILL.md](./Zotero-Obsidian-note/zotero-group-meeting-note/SKILL.md:94) 要求术语、证据、口语化复核，但没有自动验证脚本兜底，所以会出现小格式问题：额外标题、章节编号不一致、个别标点/空格不自然、过多术语中英对照等。

5. 源数据质量被直接传导到笔记
   MinerU/OCR 对表格、页眉页脚、公式、caption 的抽取质量不稳定，笔记里常出现“需人工核对”。这不是模型单独能解决的，需要把 Zotero 元数据、PDF/MinerU 结构化信息、图片-caption 映射更明确地接进流程。

**下一步优化方向**
优先把 skill 从“提示词驱动”升级成“脚本流水线 + 提示词蓝图”。

1. 增加 `validate_note.py`
   检查 H1、章节结构、路径长度、是否有 `![[Pasted image]]`、图片链接是否存在、是否有 YAML/签名、是否还有 `待核对`、图表公式是否覆盖 source-order 清单。生成失败就要求模型修复后再交付。

2. 改造路径策略
   在 `prepare_output.py` 里按“完整路径长度”预算截断，而不只是单个文件名 140 字符。对 `agent skill/<title>/<title>.md` 这种结构，建议改成短文件夹 + 可读文件名，或 `<title-short-hash>/index.md`。

3. 改造图片策略
   `collect_assets.py` 应支持 Obsidian wiki 图片 `![[...]]`，自动复制根目录 pasted images 到当前笔记 `assets/`，统一改写成 Markdown 相对路径。正文只放核心图表，非核心 MinerU 裁剪图放到 `assets/_manifest.md` 或 `assets/manifest.json`，不要默认塞进正文。

4. 强化证据清单
   不只用正则扫 Markdown，优先读取 MinerU 的结构化输出，建立 `evidence-inventory.json`：Figure/Table/Equation 编号、caption、页码、图片文件、核心/非核心、是否附录。正文的“图表公式解释”从这个清单生成并校验。

5. 增加输出模式
   默认用“组会可讲版”：控制在 150-220 行左右，突出核心证据。另设“全量审计版”：保留全部图表公式和完整资源索引。现在的成品更接近全量审计版，不太像默认组会笔记。

6. 为当前主题加专门模板
   你这批论文集中在 agent skills，可以加 `domain-agent-skill.md`：固定关注 skill 表示、skill library、检索/路由、演化、治理、安全、评估协议、负迁移和可复现性。这样会比通用 LLM 模板更锋利。
